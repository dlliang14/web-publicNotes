线上故障排查 日志分析

线上故障排查-日志分析 #

线上故障排查

工具分工 #

  • less：交互查看大日志（推荐）
  • grep：快速过滤关键词
  • awk：做统计、聚合、条件筛选
  • zgrep/zless：压缩日志直接分析

常用套路 #

less #

less /path/to/logfile.log
log show --last 10m | less
# /ERROR    向前搜索
# ?ERROR    向后搜索
# f         向前翻页
# b         向后翻页
# g/G       跳到文件开头/结尾
# F         跟随输出
# -N        显示行号
# -i        忽略大小写
# -S        不换行
# -R        显示原始控制字符（如颜色）
  • less 按需加载，适合大日志；比直接用编辑器更稳。
  • 如果环境配置了 lesspipe，压缩文件也能直接查看。
  • 进入过滤模式后可用 & 配合正则继续筛选当前内容。
  • log show --last 10m | less 适合先截取近 10 分钟日志，再进入交互式查看。
  • /ERROR 和 ?ERROR 分别是向前、向后搜索；F 类似 tail -f，适合追实时日志。

grep #

grep -iE "error|timeout|refused" app.log
grep -A 3 -B 2 "ERROR" app.log
log stream | grep -iE "proxy|kernel" --line-buffered | grep -v "debug"
  • -v 反向匹配，排除噪音行。
  • -i 忽略大小写，适合不确定日志格式的场景。
  • -w 只匹配完整单词，避免误匹配。
  • --line-buffered 适合实时流式日志，否则 grep 可能攒一批再输出。
  • -E 开启扩展正则，便于写 error|timeout|refused 这种多条件匹配。
  • -A 3 -B 2 适合报错场景，把前后文一起打出来，避免只看到单行异常。

awk #

awk '$9 >= 500 {print $0}' access.log
awk '{print $9}' access.log | sort | uniq -c | sort -nr
cat /path/to/logfile.log | awk -v RS='' '/Broken pipe/' | less
  • -F 指定列分隔符，默认空格或制表符。
  • RS 控制记录分隔符，-v RS='' 可按空行分块查看 Java 异常堆栈。
  • NF 表示列数，NR 表示记录号，$0 是整行，$1/$2/... 是具体列。
  • awk 比 grep 更适合做数值比较、字段统计和组合条件筛选。

这几个示例分别在做什么：

  • awk '$9 >= 500 {print $0}' access.log：筛出状态码或耗时位于第 9 列且大于等于 500 的记录。
  • awk '{print $9}' access.log | sort | uniq -c | sort -nr：统计第 9 列每个值出现了多少次，按频次倒序排。
  • awk -v RS='' '/Broken pipe/'：按“空行分块”匹配完整异常堆栈，而不是只匹配单行。

补充命令 #

head -n 1000 /path/to/logfile.log | grep "ERROR"
tail -n 1000 -f /path/to/logfile.log | grep "ERROR"
wc -l /path/to/logfile.log
gzip -d /path/to/logfile.log.gz -c | grep "ERROR" | less
  • 压缩日志优先用 zless、zgrep、zcat 直接分析。
  • sort 常配合 uniq -c 统计高频模式。
  • man 适合快速确认选项语义，避免在线上机器试错。

这些命令分别适合的场景：

  • head -n 1000：查看文件开头一段，适合启动失败、初始化异常。
  • tail -n 1000 -f：看文件末尾并持续追踪，适合实时报错。
  • wc -l：先估算日志规模，决定是否需要分片分析。
  • gzip -d ... -c：解压到标准输出而不改原文件，适合线上只读分析。

博客级建议 #

  • 保留“命令 + 场景 + 输出解释”三段式
  • 不只写命令，写清“为什么看这个字段”
  • 对高频故障构建可复用的过滤模板
