线上故障排查-进程与内核观测 #

线上故障排查

进程层工具 #

  • pidstat：CPU/内存/线程/上下文切换
  • ps：进程树与线程视角
  • lsof：文件、端口、句柄、删除文件占用

pidstat 常用选项 #

pidstat -t -G kubelet 1
pidstat -wv -p 1366 1
  • -u：CPU 使用率
  • -t：线程级视角
  • -G pattern：按进程名匹配
  • -r：内存使用
  • -p pid：指定 PID
  • -w：上下文切换
  • -v：线程数和文件描述符

两个示例分别适合：

  • pidstat -t -G kubelet 1：每秒观察名为 kubelet 的进程及其线程消耗，适合看线程热点。
  • pidstat -wv -p 1366 1：盯住某个 PID 的上下文切换、线程数、FD 数，适合进一步判断是否存在锁竞争或线程爆炸。

输出中 TGID 通常可视为进程 ID，TID 是线程 ID。

ps 常用方式 #

ps aux
ps -elf
ps -fC java
ps -fq 1366
ps uc -Lq 1366
top -H -i -b -d 1 -n2 -p 1
  • ps aux 看全量进程
  • ps -elf 看长格式和进程树
  • ps uc -Lq <pid> 看指定进程下的线程
  • top -H 在线程视角定位活跃线程

命令含义：

  • ps aux：适合全局扫一眼有哪些进程活着。
  • ps -elf：适合看父子关系，定位是谁拉起了谁。
  • ps -fC java：按命令名过滤，快速找 Java 进程。
  • ps -fq 1366：只看指定 PID。
  • ps uc -Lq 1366：展开指定进程的线程列表。
  • top -H -i -b -d 1 -n2 -p 1：批处理模式输出线程视角数据，常用于脚本采样或导出分析。

lsof 常用场景 #

lsof -p 1366
lsof -i :8080
lsof -a -nP -i -c kubelet
lsof | grep "xxx.log (deleted)"
  • 通过端口反查进程
  • 查看进程当前工作目录和可执行文件
  • 排查“日志已删除但磁盘空间没释放”
  • 配合 -d cwd/txt/mem 观察当前目录、二进制和内存映射文件

命令含义：

  • lsof -p 1366：查看某个进程打开了哪些文件、socket、pipe。
  • lsof -i :8080：反查谁占用了 8080 端口。
  • lsof -a -nP -i -c kubelet：按“命令名 + 网络连接”做 AND 过滤，适合查某个进程的活跃网络连接。
  • lsof +L1：找“链接数小于 1 但还被进程持有”的文件，排查删除日志后空间不释放特别常用。

内核与调用栈工具 #

  • perf：采样 CPU 热点，生成火焰图
  • bcc：场景化工具（如 biolatency、runqslower）
  • bpftrace：脚本化观测系统调用/内核事件

观测点模型 #

  • tracepoint：内核静态追踪点
  • kprobe：内核动态追踪点
  • uprobe：用户态动态追踪点
  • usdt：用户态静态追踪点，需要程序预埋探针
  • PMC/软件事件：性能计数器、缺页、中断、上下文切换等事件

常见实现体系：

  • ftrace
  • perf
  • eBPF（bcc、bpftrace）

perf 常用命令 #

perf record -F 99 -p 1366 -g -- sleep 10
perf record -F 99 -a -g -- sleep 60
perf report --stdio
perf script > out.perf
  • -F：采样频率
  • -p：指定进程
  • -g：记录调用栈
  • -a：全 CPU 采样

这几条命令的作用：

  • perf record -F 99 -p 1366 -g -- sleep 10：对单个进程采样 10 秒，适合分析“这个进程为什么吃 CPU”。
  • perf record -F 99 -a -g -- sleep 60：对整机所有 CPU 采样 60 秒，适合分析“到底是谁在吃机器”。
  • perf report --stdio：在终端直接看热点函数，不依赖图形界面。
  • perf script > out.perf：导出原始采样结果，给后续火焰图工具消费。

如果需要火焰图，可以继续做 stackcollapse 和 flamegraph 转换。

bcc / bpftrace 示例 #

runqslower-bpfcc
biolatency-bpfcc -d sda
tcpconnlat-bpfcc
gethostlatency
  • bcc 偏现成工具，适合快速套用场景。
  • bpftrace 更像 awk，适合临时写脚本追踪系统调用、网络时延或数据库请求。

这些示例各自关注：

  • runqslower-bpfcc：找调度等待过久的线程，适合 CPU 忙但又看不出谁跑满时使用。
  • biolatency-bpfcc -d sda：看磁盘 sda 的 IO 延迟分布。
  • tcpconnlat-bpfcc：看 TCP 建连耗时，适合排查网络抖动或下游连接慢。
  • gethostlatency：看 DNS 解析耗时，适合排查“接口慢但代码没变”的场景。

Java 交叉验证 #

  • Java 场景常配合 jstack、jmap、arthas。
  • 如果要做 Java 火焰图，通常需要完整调用栈支持，例如保留帧指针。

何时用哪类工具 #

  1. 先用 top/pidstat 锁定可疑进程。
  2. 再用 perf 看热点函数与调用链。
  3. 怀疑系统调用/内核瓶颈时，用 bpftrace/bcc。
  4. Java 场景配合 jstack/jmap 做用户态交叉验证。

经验 #

  • 先粗后细，避免上来就重型观测。
  • 线上观测控制采样时长，降低扰动。
  • 输出要落地为“改进动作”，而不是止于截图。
