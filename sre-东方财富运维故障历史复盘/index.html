sre-东方财富运维故障历史复盘 #

sre

双控冗余配置失败 增加流量监控 #

  • centos 6 和 7 配置不同，双控冗余配置失败

  • 只监控了端口active状态，没有监控数据流量，导致故障发生时无法及时发现。

  • 重启orcale数据库后，切换到左边控制器，恢复服务。

  • 故障时间：2021/7/30非高峰时段9:55-10:15，6、7、8分区部分用户约20分钟无法交易。

  • 原因：EMC存储的右下角控制器发生硬件故障，导致交易数据库down。自动切换失效，手动切换控制器恢复时间较长。

  • 故障总结：对存储硬件设备的应急演练不够，监控不到位。现已增加控制器的流量监控和报警，可以杜绝此类故障再次发生。

盘中hosts更新 加强变更管理 #

  • 开发感觉服务不通，需要调整hosts。
  • 直接修改了生产环境的hosts，不该通的服务通了，导致部分用户访问异常。

外部依赖单点 增加备份链路 #

外部系统只提供单链路或单设备，内部虽冗余仍可能整体不可用。

治理要点：

  • 外部依赖纳入同等冗余审计。
  • 对关键依赖建立替代路径与降级路径。
  • 后续又要了一个备份链路，2个单网口。

冷备切换不熟练 单点服务要做好切换演练 #

  • 因为合规要求，踢人服务只能单台。
  • 有冷备，但是ip不一致，切换后没有更新，导致服务不可用。
  • 后续新增放宽逻辑，允许先不踢人，后续收盘再踢人。
  • 单点服务要做好切换演练，确保切换流程熟练，切换后相关配置也要及时更新。

配置错误 A角配置B角复查 #

  • 因为历史原因，收费和免费的端口是一样的，人为配置错误。

流量监控 #

  • 流量突然增大，怎么办
  • net-flow 分析流量，通过五元组快速定位哪个业务把专线流量打高。
