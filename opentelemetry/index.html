opentelemetry collector extension #

  • health_check：提供健康检查接口，监控 Collector 的运行状态。
  • pprof：暴露性能分析接口，帮助诊断 Collector 的性能问题。
  • zpages：提供一个 Web 界面，展示 Collector 的运行状态和性能指标，方便调试和监控。

opentrace opencensus opentelemetry 区别 #

OpenTracing (老大哥：定规矩的) #

  • 初衷：当时分布式追踪工具（Jaeger, Zipkin）层出不穷，代码不通用。OpenTracing 制定了一套通用的 API 标准。
  • 缺点：它只是一张“纸”，没有实际干活的代码（SDK）。你得自己找兼容这个标准的厂商实现。而且它完全不支持 Metrics（指标）。

OpenCensus (二哥：带装备的) #

  • 初衷：由 Google 发起，它不仅定义了标准，还直接给出了 SDK（库）。你集成到代码里就能直接把数据发给后端。
  • 特点：同时支持 Tracing 和 Metrics。
  • 缺点：不兼容 OpenTracing，导致开发者被迫面临“二选一”的难题。

OpenTelemetry (终极合体：我全都要) #

  • 背景：为了结束行业分裂，OpenTracing 和 OpenCensus 在 2019 年宣布合并。
  • 地位：它是云原生计算基金会（CNCF）中活跃度仅次于 Kubernetes 的项目。
  • 优势：
  • 三位一体：一套 SDK 搞定 Tracing（链路）、Metrics（指标）、Logging（日志）。
  • 向前兼容：提供桥接层，兼容老代码里的 OpenTracing/OpenCensus 埋点。
  • 语言支持极广：Java, Go, Python, JS, C++, Rust 等都有官方支持。
  • 解耦：引入了 Collector（收集器），你的应用只需要把数据发给 Collector，由 Collector 决定发往 VictoriaMetrics、Jaeger 还是 Elasticsearch。

监控项 #

自研告警系统 #

  • 存活探针
  • 指标告警
  • 告警消息丢弃监控(防止推送太频繁)

业务监控 #

  • 下单成功数
  • 登录人数

应用服务监控 #

  • 流量分析：比如调用次数最多的接口、请求分布等。
  • 四大黄金指标：请求速率（RPS）、错误率、请求延迟（P95/P99）和资源使用率。
  • 数据库：查询次数、慢查询数、连接数等。
  • go 运行时：go sched lantency、gc duration、goroutine 数量、内存使用等。

trace 跟踪 #

  • 对慢sql，长事务进行跟踪
  • 对慢redis进行跟踪
  • 对慢http接口进行跟踪
  • 对慢grpc接口进行跟踪
  • 对慢三方http接口进行跟踪

性能监控的原则 #

不要高基数 #

  1. 标签越多，重复性越低，基数越高，导致聚合分析越慢
  2. 通过tsdb-status 监控基数，过高需要优化
    1. Label Names with Value Count：关注的是单个标签的多样性
    2. series-metrics 指标的序列数量，同一个指标里的标签组合只要不一样，就是一个新序列
    3. label-memory 监控标签占用的内存
    4. Series Count (by Label Value Pairs)：关注的是组合后的最终结果

指标要能发现和定位问题 #

按监控分层维度建立指标 #

  • 机器
  • 中间件
  • 应用维度 qps
  • 业务维度 订单量，登录人数

不要企图一开始就监控十全十美 #
